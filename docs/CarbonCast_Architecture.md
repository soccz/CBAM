# CarbonCast 아키텍처 문서

> ESG TradeGuard 플랫폼의 CBAM 비용 분석 모듈
> CBAM: EU의 탄소가격 격차 조정 메커니즘 (탄소 관세에 준하는 가격 조정)
> 최종 업데이트: 2026.03.25

---

## 1. 시스템 개요

### 1.1 CarbonCast가 하는 것

```
기업이 배출 데이터 제출
  → [모듈 1] AI 품질 스코어링 (데이터 신뢰성 사전 검증)
  → [모듈 2] 블록체인 기록 (검증 이력 변조 불가 보관)
  → [모듈 3] CBAM 비용 시뮬레이션 (연도별 비용 경로 산출)
```

### 1.2 ESG TradeGuard 내 위치

```
Step 1: 서류 업로드 (READIT OCR)        ← 팀원 담당
Step 2: OCR 파싱 (HS코드, 수량)          ← 팀원 담당
Step 3: 규제 매칭 (CBAM/EUDR)           ← 팀원 담당
Step 4: ★ CarbonCast (품질검증→인증→비용) ← 이 모듈
Step 5: 위성 NDVI (환경 검증)            ← 팀원 담당
Step 6: 최종 리포트                      ← 팀원 담당
```

### 1.3 두 가지 사용 트랙

| Track | 대상 | CarbonCast가 하는 것 |
|:------|:-----|:-------------------|
| **A — 데이터 보유 기업** | K-ETS 대상, 대기업 | 모듈 1(품질 검증) → 모듈 2(블록체인) → 모듈 3(비용 시뮬레이션) |
| **B — 데이터 미보유 기업** | 중소 수출기업 | 모듈 3만 사용: "기본값 vs 실측 시 얼마를 아끼는지" → 데이터 확보 동기 제공 |

## 2. 모듈 1: AI 품질 스코어링

### 2.1 역할

- EU accredited verifier 검증 **전 단계**에서의 **사전 오류 스크리닝**
- EU 공식 검증을 대체하지 않음 (ISO 14065 인증기관만 공식 검증 수행)
- 0~100 품질 스코어로 즉시 판단

### 2.2 입력/출력

```
[입력]
├── 업종 (철강 고로/전기로, 알루미늄, 시멘트, 비료, 수소)
├── 배출계수 (tCO₂/톤 제품)
├── 에너지 사용량 (GJ/톤 제품)
└── 연간 생산량 (톤)

[출력]
├── 품질 스코어 (0~100)
├── 판단 (✓ 신뢰 가능 / △ 추가 확인 / ✗ 재확인 필요)
├── 체크 항목별 상세 (4개 기준)
├── 업종별 분포 내 위치 시각화
└── EU 기본값 대비 절감 가능액 (있는 경우)
```

### 2.3 스코어링 알고리즘

4가지 기준의 가중 합산:

| # | 판단 기준 | 방법 | 감점 범위 | 데이터 소스 |
|:--|:---------|:-----|:---------|:-----------|
| 1 | **배출계수 범위** | z-score. 범위 밖 -50, 범위 안 max(30, zEf×10) | 0~50점 | K-ETS 명세서 약 800여 개 사업장 |
| 2 | **에너지 범위** | z-score. 범위 밖 -20, 범위 안 min(15, zEn×5) | 0~20점 | K-ETS 명세서 |
| 3 | **에너지-배출 정합성** | `ratio = EF / (energy × mean/energy_mean)`. 0.5~1.8 밖이면 -20 | 0~20점 | 물리법칙 기반 |
| 4 | **생산 규모 합리성** | 최소 기준 (100톤). 향후 DART 재무 연동 시 매출 대비 배출 집약도 비교로 확장 예정 | 0~5점 | 규칙 기반 |

```
score = 100 - (배출계수 감점) - (에너지 감점) - (정합성 감점) - (생산량 감점)
score = max(0, min(100, round(score)))
```

**판단 기준:**
- 80점 이상: ✓ 신뢰 가능
- 60~79점: △ 추가 확인 권장
- 60점 미만: ✗ 데이터 재확인 필요

### 2.4 업종별 파라미터

> 출처 확인 상태: ✓ 확인됨 / ※ 미확인 예시값

| 업종 | 평균 EF | σ | 물리 범위 | EU 기본값 | 벤치마크 | 출처 상태 |
|:-----|:--------|:--|:---------|:----------|:---------|:---------|
| 철강 고로 (BF-BOF) | 2.05 | 0.15 | 1.6~2.5 | 3.00 | 1.370 | ✓ 기본값: SteelRadar/IR 2025/2621. 벤치마크: Eurometal/S&P Global |
| 철강 전기로 (Scrap EAF) | 0.45 | 0.10 | 0.2~0.9 | 3.00 | 0.072 | ✓ 벤치마크: Eurometal |
| 알루미늄 | 1.60 | 0.25 | 0.8~2.5 | 4.90 | 1.514 | ※ 기본값: Coolset 예시. 벤치마크: 미확인 |
| 시멘트 (클링커) | 0.64 | 0.06 | 0.45~0.85 | 0.87 | 0.666 | ✓ 벤치마크: Global Cement/S&P Global |
| 비료 (암모니아) | 2.30 | 0.35 | 1.5~3.5 | 3.60 | 1.96 | ※ 기본값/벤치마크: 미확인 예시값 |
| 수소 | 9.00 | 1.50 | 5~13 | 12.0 | 8.85 | ※ 기본값/벤치마크: 미확인 예시값 |

### 2.5 기술 스택

| 항목 | 선택 | 이유 |
|:-----|:-----|:-----|
| 이상치 탐지 | Isolation Forest + KDE | 다변량 이상치 탐지 + 해석 가능 |
| 물리적 범위 체크 | 규칙 기반 (하한/상한) | 공정별 물리적 한계는 규칙이 확실 |
| 학습 데이터 | K-ETS 명세서 (data.go.kr/15053947) | 약 800여 개 사업장, 공개 데이터 |
| 스코어링 | 0~100 정규화 | Isolation Forest anomaly score + 물리 체크 가중 합산 |

### 2.6 왜 ML이 필요한가

- 단순 상한·하한 규칙으로는 "고로인데 배출계수가 0.8" 같은 명백한 케이스만 잡힘
- "고로인데 에너지 사용량이 전기로 수준이면서 배출계수만 살짝 낮다" 같은 **다변량 패턴**은 규칙 조합이 기하급수적으로 증가
- ML(Isolation Forest)은 이 다변량 이상치를 자동 학습

### 2.7 한계

- 프로토타입의 스코어링은 z-score 기반 규칙에 가까움 (진정한 Isolation Forest 학습 결과가 아님)
- 실 데이터(K-ETS 명세서) 적용 시 모델 정확도 검증 필요
- 배출계수 분포를 정규분포로 가정했으나, 실제로는 bimodal(고로/전기로 혼합) 가능

## 3. 모듈 2: 블록체인 기록

### 3.1 역할

- 품질 스코어링 통과(60점 이상)한 배출 데이터의 **감사 추적(audit trail)**
- **CBAM 공식 레지스트리(EU 중앙 시스템)와는 별도의 민간 데이터 인프라**
- 한국 수출기업 ↔ EU 수입자 ↔ 검증기관 사이의 데이터 변경 불가성(immutability) + 타임스탬프

### 3.2 데이터 구조

```
[온체인 기록]
├── txHash: SHA-256 해시
├── block: 블록 번호
├── timestamp: ISO 8601
├── company_id: 익명화된 식별자 (HANA-XXXXXX)
├── cn_code: 제품 CN코드 (예: 7207)
├── sector: 업종명
├── emission_factor: 배출계수 (tCO₂/t)
├── quality_score: 품질 스코어 (/100)
└── verifier: HANA_CARBONCAST_v2

[오프체인 (하나금융 DB)]
├── 원본 데이터 전체
├── 에너지 사용량, 생산량 등 상세
└── 기업 식별 정보
```

### 3.3 해시 생성

```javascript
// Web Crypto API 표준 구현
async function sha256(msg) {
  const buf = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(msg));
  return Array.from(new Uint8Array(buf)).map(b => b.toString(16).padStart(2,'0')).join('');
}

// payload = JSON.stringify({ company_id, cn_code, sector, emission_factor, ... })
const hash = await sha256(payload);
```

### 3.4 변조 감지

```
원본 데이터 → SHA-256 → 해시A (온체인 저장)
                  ↓
나중에 검증 시: 원본 데이터 → SHA-256 → 해시B
                  ↓
해시A === 해시B ? "변조 없음" : "변조 감지"
```

### 3.5 기술 스택

| 항목 | 프로토타입 | 실전 |
|:-----|:----------|:-----|
| 네트워크 | Ethereum Sepolia 테스트넷 | Hyperledger Fabric (허가형) |
| 스마트 컨트랙트 | Solidity (EVM 호환) | Hyperledger Chaincode |
| 해시 | SHA-256 (Web Crypto API) | 동일 |
| 데이터 저장 | 온체인: 해시+메타, 오프체인: 로컬 | 온체인: 해시+메타, 오프체인: 하나금융 DB |

### 3.6 접근 제어

- 품질 스코어 60점 미만 → 블록체인 기록 **거부**
- 블록체인 기록은 검증 통과 시에만 실행

### 3.7 한계 및 주의사항

- EU가 인정하는 검증은 accredited verifier의 서명이 들어간 검증보고서(verification report)
- 블록체인 해시는 **법적 효력 없음**. 감사 추적용 보조 인프라
- "CBAM 공식 레지스트리와 연동" 뉘앙스 절대 지양
- EU DPP(2027/2028년경 단계적 의무화 예상) 연계 시 참조 로그로 활용 가능하나, 공식 인프라는 아님

## 4. 모듈 3: CBAM 비용 시뮬레이션

### 4.1 역할

- 검증된 배출 데이터(또는 EU 기본값) 기반 **2026~2034년 연도별 CBAM 비용 경로 산출**
- 4개 NGFS 시나리오 동시 비교
- 기본값 vs 실측 절감액 계산

### 4.2 CBAM 비용 공식

```
CBAM 비용 = 수출량 × [배출계수 − 벤치마크 × (1 − 무상할당 폐지율)] × EUA 가격 × 환율
```

출처: EU Regulation 2023/956, Art. 7; 계산 방법론 참고: CarbonChain CBAM Exposure Methodology (민간 방법론)

**공식 해설:**
- `무상할당 폐지율`이 올라갈수록 → 벤치마크 공제가 줄어들어 → CBAM 부담 증가
- 2026년(폐지율 2.5%): 벤치마크의 97.5%가 공제 → 부담 적음
- 2034년(폐지율 100%): 공제 0% → 배출계수 전체에 대해 비용 발생

### 4.3 JS 구현

```javascript
// 정확한 공식 구현
const phaseInRate = 1 - PHASE_IN[year];  // 무상할당 잔여율
const liable = useEf - benchmark * phaseInRate;
const cost = volume * Math.max(0, liable) * ETS_PRICES[scenario][year] * fx / 1e8; // 억원
```

**기본값 적용 시 연도별 마크업 반영:**
```javascript
const getMarkup = (y) => y <= 2026 ? s.markup_2026 : y <= 2027 ? s.markup_2027 : s.markup_2028;
// 2026: +10%, 2027: +20%, 2028~: +30%
const useEf = dataType === 'default' ? eu_default * getMarkup(year) : actualEf;
```

### 4.4 EU ETS 무상할당 폐지율 (Phase-in)

| 연도 | 2026 | 2027 | 2028 | 2029 | 2030 | 2031 | 2032 | 2033 | 2034 |
|:-----|:-----|:-----|:-----|:-----|:-----|:-----|:-----|:-----|:-----|
| 폐지율 | 2.5% | 5% | 10% | 22.5% | 48.5% | 61% | 73.5% | 86% | 100% |

출처: EU Regulation 2023/956, Annex IV (EU ETS 무상할당 단계적 폐지 스케줄)

| 연도 | 일반 업종 (철강, 알루미늄, 시멘트) | 비료 |

### 4.7 K-ETS 공제 처리

- 이론적으로 수출국에서 지불한 탄소 가격은 CBAM에서 공제 가능 (EU Regulation 2023/956, Art. 9)
- 한국은 K-ETS 무상할당 비율이 높아 실질 납부액이 적음
- **프로토타입에서는 K-ETS 공제를 미반영 (보수적 추정)** → 비용 과대 추정 방향
- 실전 전환 시: K-ETS 유상매입분 데이터 연동 필요

### 4.8 기술 스택

| 항목 | 선택 | 이유 |
|:-----|:-----|:-----|
| 비용 공식 | EU Regulation 2023/956 기반 | 수식 기반. AI가 아님. 정확해야 하므로. |
| 시나리오 | NGFS Phase V, 4개 경로 | TCFD/ISSB 권고 부합. IIASA Explorer 공개. |
| 대시보드 | Streamlit + Plotly (실전) / HTML Canvas (프로토타입) | 인터랙티브 슬라이더 |
| EU ETS 가격 데이터 | Yahoo Finance (ICE EUA Futures) | 무료, 지연 시세. 시뮬레이션에 충분한 해상도. |

### 4.9 한계

- NGFS 가격 경로는 "근사"이며, 정확한 NGFS Phase V 수치는 IIASA Explorer에서 직접 확인 필요
- 간접배출(Scope 2) 미반영 — 알루미늄은 간접배출이 CBAM 대상이나 현재 모델에서 미포함
- 복합 제품(downstream)의 내재 탄소 추적 미지원

## 5. 데이터 흐름

### 5.1 ESG TradeGuard → CarbonCast

```
[Step 3에서 받는 것]
  HS코드, 수출국, 수량, 금액
    ↓
  CBAM 대상 여부 확인 + 제품별 기본값 매핑

[기업 추가 입력 (Track A)]
  배출계수, 공정 유형, 에너지 사용량
    ↓
  모듈 1: AI 품질 스코어링
    ↓ (60점 이상)
  모듈 2: 블록체인 기록
    ↓
  모듈 3: CBAM 비용 시뮬레이션

[기업 추가 입력 없음 (Track B)]
  업종, 수출량만
    ↓
  모듈 3: 기본값 적용 → "실측 확보 시 절감액" 제시
```

### 5.2 CarbonCast → Step 6 (최종 리포트)

```
[CarbonCast 출력]
├── 품질 스코어 + 상세 체크 결과
├── 블록체인 인증 해시 + 타임스탬프
├── 연도별 CBAM 비용 경로 (4개 시나리오)
├── 기본값 vs 실측 절감액
└── → Step 6 리포트에 포함
```


## 8. 알려진 한계 및 향후 과제

### 8.1 제도적 한계

- CBAM 신고 의무는 EU 수입자에게 있음 → CarbonCast는 한국 수출기업이 데이터를 "제공"하는 도구
- AI 품질 스코어링은 EU 공식 검증(accredited verifier)을 대체하지 않음
- 블록체인 기록은 EU CBAM 레지스트리와 별도의 민간 인프라
- K-ETS 공제 미반영 → 실전에서는 유상매입분 데이터 연동 필요

### 8.2 기술적 한계

- 알루미늄 간접배출(Scope 2) 미포함 → CBAM에서는 알루미늄 간접배출 대상
- 복합 제품(downstream) 내재 탄소 추적 미지원 → 2028년 CBAM 확대 시 필요
- 프로토타입 스코어링은 z-score 기반 → 실전에서는 Isolation Forest 학습 필요
- 배출계수 분포를 정규분포로 가정 → 실제는 bimodal 가능
- crypto.subtle은 HTTPS/localhost에서만 동작 → HTTP 환경 미지원
